关键字

local

local关键字用于声明一个局部变量，其作用域仅限于当前函数。这意味着这个变量只在当前函数内部可见和可用，并且不会影响外部作用域的同名变量。

local关键字的主要作用是：

  1. 避免变量污染: 在函数内部，您可能会使用与外部作用域相同名称的变量，但是通过使用local声明局部变量，可以确保函数内部的操作不会影响到外部同名变量。这有助于减少命名冲突和不必要的副作用。
  2. 作用域控制: local允许您在函数内部创建一个独立的作用域，使得函数内部的变量不会泄漏到全局作用域。这有助于编写更清晰和可维护的代码，因为您可以更容易地理解和跟踪变量的生命周期。

if

if 语句后面可以使用单中括号 [ ] 或双中括号 [[ ]]，它们有一些区别

  1. 单中括号 [ ]：
    • 单中括号是旧式的测试结构，仍然被支持，但不够灵活。
    • 单中括号要求在内部使用空格来分隔参数和运算符，例如 [ "$var" -eq 5 ]。
    • 单中括号在某些情况下可能需要引号来防止意外的参数展开和拆分。
  2. 双中括号 [[ ]]：
    • 双中括号是Bash中的扩展测试结构，通常比单中括号更强大和灵活。
    • 双中括号不需要在内部使用引号来保护变量，例如 [[ $var -eq 5 ]]。
    • 双中括号支持更多的条件测试，例如 [[ $var == "string" ]] 和 [[ $var =~ pattern ]]。
    • 双中括号在逻辑表达式中更自然，例如 [[ $var1 -eq 5 && $var2 == "string" ]]。

双中括号 [[ ]] 更加强大、灵活且易于使用，它是在Bash中推荐使用的测试结构。但是，单中括号 [ ] 仍然在某些情况下有用，特别是在编写与POSIX兼容的脚本时或在其他Shell中运行时。

输出重定向

  • >：将标准输出重定向到指定文件中，覆盖原有内容。
  • >>：将标准输出重定向到指定文件中，追加到原有内容的末尾。
  • < ：符号是输入重定向的一种形式。它从文件中读取输入并将其传递给命令
  • <<<：将一个字符串作为输入传递给命令。
  • &>>：将标准输出和标准错误输出都重定向到指定文件中，追加到原有内容的末尾。

eval

eval 是一个内建的shell命令，它用来执行由参数组成的命令。在执行之前，会先对参数进行一次命令行展开，这意味着可以先对参数进行变量替换、命令替换等操作。

通常情况下，输入到shell的命令只会经过一次展开就直接执行了。但是使用 eval 可以使得命令行被二次展开，第一次展开生成新的命令行，然后这个新的命令行再次被展开并执行。

总结以上，将一个命令放入eval中，按下回车后，执行eval后的代码生成的结果，再对结果进行执行。

后台命令脱离终端运行

系统更新时如果退出终端或者退出登录就会使更新系统程序中断，这时就需要脱离终端运行且关闭终端继续运行。中断原因是，当用户注销（logout）或者网络断开时，终端会收到 HUP（hangup）信号从而关闭其所有子进程。因此，我们的解决办法就有两种途径：要么让进程忽略 HUP 信号，要么让进程运行在新的会话里从而成为不属于此终端的子进程。

参考：

CSDN-koberonaldo24：linux后台运行、挂起、恢复进程相关命令

nohub

nohup的用途就是让提交的命令忽略 hangup 信号。nohup 的使用是十分方便的，只需在要处理的命令前加上 nohup 即可，标准输出(终端不打印)和标准错误缺省会被重定向到 nohup.out 文件中(未实验)，nohub就相当于一个保护进程，加上&才可以将进程放入后台运行(ps -ef可见)，如果不使用onhub只在需要运行于后台的命令结尾加上&，就会使命令的打印在当前终端打印输出，并且关闭终端会使此命令中止运行，但不会影响新敲入的命令的执行。

nohub命令的使用：

  1. nohup图形化命令窗口点击关闭不会停止当前工作台的所有后台运行进程。
  2. nohup远程登录直接可以退出xshell等远程连接软件，后台程序不会退出
  3. 若要关闭，可使用ps -ef | grep <命令关键词>，搜索到PID,使用kill杀掉此后台进程。

如果您在KDE图形化的命令行窗口中运行的程序没有使用nohup等工具将其与终端分离，那么当您关闭终端窗口时，该程序将收到HUP信号并被终止。这是因为当您关闭终端窗口时，终端会将HUP信号发送给所有在此终端中运行的进程。

如果您希望运行的程序在终端关闭后继续运行而不受HUP信号的影响，可以在执行命令时使用nohup命令或将其与screen等工具一起使用，这将使程序与终端分离，并且不会受到HUP信号的影响。

setsid

如果我们换个角度思考，如果我们的进程不属于接受 HUP 信号的终端的子进程，那么自然也就不会受到 HUP 信号的影响了。setsid 就能帮助我们做到这一点。 setsid 的使用也是非常方便的，也只需在要处理的命令前加上 setsid 即可。

setsid命令和nohup使用方法一样，直接执行例如以下

nohup和setsid都可以让进程脱离终端、不受HUP信号的影响，并在后台独立运行。它们的主要区别在于：

  1. 使用方式：nohup是在命令前加上nohup命令，例如：nohup command &；而setsid是通过系统调用setsid来创建一个新的会话和进程组。
  2. 进程控制：nohup命令不会创建新的进程组，因此进程仍然受到原来的进程组的控制，如果原来的进程组终止，该进程也会终止；而setsid创建一个新的进程组，进程不再受到原来的进程组的控制，即使原来的进程组终止，该进程也不会终止。
  3. 输出重定向：nohup命令会将标准输出和标准错误输出重定向到指定的文件中；而setsid不会自动将输出重定向，需要手动进行输出重定向。

因此，如果您需要在后台运行一个进程，并且希望该进程不受HUP信号的影响，可以使用nohup或setsid。如果您需要控制进程组并将输出重定向到文件中，可以使用nohup；如果您需要在新的会话和进程组中运行进程，并手动进行输出重定向，可以使用setsid。
